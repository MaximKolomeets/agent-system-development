# FILE_COMMENTING_STANDARD

## Назначение

Этот стандарт закрепляет правило русских комментариев в создаваемых и изменяемых файлах methodology repository и target repositories.

Русские комментарии обязательны для нужных строк и блоков, если формат файла поддерживает комментарии.

Комментарии нужны не для украшения, а для безопасной передачи смысла: что делает строка или блок и зачем он нужен.

## Где комментарии обязательны

Комментарии обязательны для нужных строк и блоков в:

- scripts;
- workflow files;
- automation files;
- config-like files;
- template files;
- технических примерах команд;
- блоках, которые меняют поведение CI, Docker, Git, security checks или release workflow.

## Что считать нужной строкой

Нужной считается строка или блок, которые:

- изменяют файлы;
- удаляют файлы;
- перемещают файлы;
- делают commit, push, merge или rebase;
- работают с remote repository;
- создают ветки;
- запускают проверки безопасности;
- читают или фильтруют sensitive patterns;
- меняют настройки CI, Docker или workflow;
- запускают внешние инструменты;
- содержат неочевидную бизнес-логику;
- задают архитектурное правило;
- могут привести к потере данных при ошибке.

Очевидные строки не нужно комментировать, если комментарий создает шум.

## Как комментировать скрипты

В скриптах комментарий должен стоять рядом с важной строкой или перед важным блоком.

Комментарий должен отвечать на два вопроса:

- что делает строка или блок;
- зачем это нужно в контексте задачи.

Пример:

```powershell
# Проверяем рабочее дерево до переключения веток, чтобы не потерять локальные изменения.
git status --short

# Подтягиваем только fast-forward, чтобы не создавать merge commit в рабочей ветке.
git pull --ff-only origin developer
```

## Как комментировать YAML, workflow и config-like files

Если формат поддерживает комментарии, важные safety и behavior blocks должны иметь русский комментарий.

Пример:

```yaml
# Запускаем проверку на Pull Request, чтобы forbidden tracked files не попадали в protected branches.
on:
  pull_request:
```

Если комментарии могут сломать формат или ухудшить читаемость, пояснение переносится в соседнюю документацию.

## Форматы, где комментарии запрещены

В форматах без поддержки комментариев, например JSON, комментарии не вставляются в сам файл.

Вместо этого используются:

- соседний README;
- schema descriptions;
- отдельная документация;
- пояснение рядом с template.

## Как не превращать комментарии в шум

Не нужно комментировать каждую строку.

Комментарий не нужен, если строка очевидна и не несет safety, state-changing или архитектурного смысла.

Комментарий нужен, если без него следующий `engine` или человек может не понять риск, намерение или ограничение.

## Русский язык комментариев

Комментарии внутри файлов должны быть на русском языке.

Английский сохраняется для commands, code identifiers, config keys, package names, API names, vendor/tool names, branch names и file names.

## Safety comments

Комментарии не должны содержать:

- secrets;
- private data;
- client data;
- customer data;
- private repository URL;
- internal code names;
- реальные tokens, passwords или API keys.

Если пример показывает sensitive grep, он должен быть filename-only и не должен печатать matching lines.

## Примеры

Хороший комментарий:

```powershell
# Выполняем filename-only sensitive grep, чтобы проверить риск секретов без вывода matching lines.
git grep -I -l -i -E "token|password|secret|api_key|apikey|credential|пароль|токен" --
```

Избыточный комментарий:

```powershell
# Печатаем текущую ветку.
git branch --show-current
```

Такой комментарий допустим в обучающем terminal block, но в обычном скрипте он чаще всего шумит.
