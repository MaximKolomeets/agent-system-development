# SOURCE_CONSUMERS

Реестр downstream-проектов (потребителей), которым нужен Source-снапшот методологии `agent-system-development`.

Используется правилом «Source-reminder» (канон — `docs/agent-system/templates/TASK_HEADER_COMMON.md`): если задача меняла методологию/каноны, исполнитель обновляет source-снапшот и в RESULT и блоке «Передача» явно перечисляет потребителей из этого реестра строкой «Обновить Source-снапшот в проектах: …».

Ответственная роль за синхронизацию — `source-steward` (канон ролей — `docs/agent-system/ROLE_MODEL.md`).

## Принципы

- Source of truth — GitHub-файлы methodology repository. Снапшоты в downstream-проектах являются derived context (политика снапшота — `docs/agent-system/source/README.md`).
- Реестр перечисляет, **кому** нужен снапшот; синхронизация выполняется в каждом потребителе по его branch-модели через PR, а не прямой правкой.
- Реестр не содержит private data, credentials, tokens и внутренних кодовых имён; только публично известные repository-указатели.
- Запись добавляется/обновляется, когда у потребителя появляется/меняется зависимость от методологического снапшота.

## Реестр потребителей

| Потребитель | Repository | Что потребляет | Канал синхронизации | Ответственная роль | Примечания |
|---|---|---|---|---|---|
| verification | github.com/MaximKolomeets/verification | Source-снапшот методологии: role-agnostic шаблоны задач/review, каноны (`TASK_HEADER_COMMON`, `ROLE_MODEL`, governance) | PR в branch-модели потребителя | source-steward | Засеяно задачей `METH-EXECUTOR-AGNOSTIC-V1`. |

## Как обновлять реестр

- Новый потребитель — добавить строку с repository-указателем, перечнем потребляемого, каналом синхронизации и ответственной ролью.
- Потребитель больше не зависит от снапшота — пометить строку как архивную в примечании (не удалять историю без отдельного решения пользователя).
